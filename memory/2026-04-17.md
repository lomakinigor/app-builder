## Сессия 2
Тема: T-019 — Проверка sound notification playback path

### Сделано

- Создан `tests/e2e/helpers/audioMonitor.ts`:
  - `injectAudioMonitor(page)` — патчит `AudioContext.prototype.createOscillator` через `addInitScript`, каждый `OscillatorNode.start()` пишет в `window.__audioPlayLog`
  - `getPlayedSounds(page)`, `clearPlayedSounds(page)`, `waitForSoundAttempt(page, ms)`
  - Пишет `{ timestamp, type: 'oscillator-start', contextState }` — contextState позволяет диагностировать suspended/running

- Создан `tests/e2e/sound-notifications.spec.ts`:
  - SOUND-001: sound ON → click "Проверить звук" → oscillator-start fired ✅
  - SOUND-002: sound OFF → preview button скрыт + log пустой ✅
  - SOUND-003: toggle OFF→ON round-trip → preview fires ✅

- Результаты: 3/3 на 4 прогонах, стабильно

### Решения / Root cause

**Нет production bug.** Код корректен.

Диагноз:
- В Playwright Chromium: `contextState = 'running'` — AudioContext стартует в running state потому что Playwright не имеет autoplay ограничений
- В реальном браузере: AudioContext создаётся ПОСЛЕ user gesture (пользователь кликнул "Generate" / "Parse"), поэтому Chrome создаёт его в running state
- Код в `playBeep()` правильно обрабатывает suspended state через `resume().then(doBeep)`
- `settingsStore.onRehydrateStorage` корректно синкает `soundEnabled` при hydration

**"Звука нет"** = observation в тестовой среде (нет реальных speakers/headphones), не production bug.
